# SPEC — MMA Fight Analysis & Fighter Skill Tree

## 1. Goal
Take footage from an MMA fight (primarily UFC) and automatically:
1. Catalogue each fighter's **stance** (orthodox / southpaw / switching).
2. Detect and classify the **strikes** they throw (punches and kicks).
3. Aggregate the above into a **per-fighter skill tree** summarising techniques, tendencies, and favored combinations.

This is a personal project. Prioritise a working end-to-end MVP over completeness.

## 2. Architecture (fixed — 5 stages)
The system is a multi-stage pipeline, not a single model. Each stage is independently testable and swappable.

1. **Detection & tracking** — find each fighter per frame, keep consistent IDs across frames.
2. **Pose estimation** — per-fighter 2D skeleton (COCO-17 keypoints). Core feature representation for everything downstream.
3. **Stance classification** — geometric rule over pose keypoints, smoothed over a temporal window.
4. **Strike / action recognition** — temporal action detection over pose sequences (the hard ML core).
5. **Aggregation & skill tree** — tally techniques per fighter, derive a structured skill-tree summary + visualization.

## 3. Dataset reality (read this before assuming any ground truth exists)
**Source:** Mendeley `c456bnk8bm`, Hasan Faisal (Bahria University). License CC BY-NC-SA 4.0.
- **V1** (DOI 10.17632/c456bnk8bm.1): 5,106 images, 20 UFC fights, 640×640, single class `fighter`, YOLOv8 bbox format.
- **V2** (DOI 10.17632/c456bnk8bm.2): same images + 17-point COCO keypoints per fighter, YOLO-pose format. Splits: train 3,635 / valid 980 / test 491. **Use V2** (superset of V1).

**What the dataset covers:** Stage 1 (detection) and Stage 2 (pose) only.

**What the dataset does NOT contain — do not hallucinate these:**
- **No temporal data.** Independent extracted frames, not ordered video. Strike recognition is inherently temporal and cannot be learned from these stills.
- **No strike / action labels.** Zero ground truth for kicks/punches.
- **No stance labels.** Orthodox/southpaw was only a *selection* criterion, not an annotation.
- **No fighter identity.** Both fighters share one class `fighter`; A vs B is not distinguished.
- **Keypoints are pseudo-labels** auto-generated by YOLOv11x-pose — approximate, especially under occlusion.
- **Stand-up only.** Ground grappling and clinch deliberately excluded.

**Implication:** the dataset gets Stages 1–2 done. Stages 3–5 require sourcing actual *video* and creating *our own labels*.

### 3a. Label format vs. inference format — DO NOT MIX (learned the hard way)
The dataset label files and live model inference use **different conventions** for two things. The pipeline must speak the *inference* format, since that's what video produces.
- **Visibility/confidence:** label files store an integer flag `{0,1,2}`; inference returns a **float confidence `0.0–1.0`**. Use `min_vis=0.5`, never `min_vis=2` (which can never be satisfied by inference output → everything reads as absent).
- **Coordinates:** label files are **normalized `0–1`**; inference returns **pixel coordinates**. Never compare across the two. The stance rule only compares values *relative to each other*, so it is unit-agnostic, but the overlay and any absolute thresholds must use pixels.

## 4. Strike taxonomy (labeling target)
Start with a small MVP subset, expand later.

**MVP classes (6):** `jab`, `cross`, `hook`, `leg_kick`, `head_kick`, `idle/none`

**Full target taxonomy:**
- Punches: `jab`, `cross`, `hook`, `uppercut`, `overhand`
- Kicks: `leg_kick` (low), `body_kick`, `head_kick` (high), `front_kick/teep`
- Other strikes: `knee`, `elbow`
- Non-strike: `idle`, `clinch`, `takedown_attempt`

Keep label definitions in this file as they get pinned down so annotation stays consistent.

## 5. Stance definition (rule, not a learned model)
Stance = which body side is bladed toward the opponent. Orthodox = left side leads; southpaw = right side leads. Derived per-frame from pose, then smoothed over time. Output is a **distribution per fighter over the fight**, not a single static label.

**Current implementation — lead-foot rule over ankles (`frame_stance`). This is the WORKING method.**
- **Forward axis** = direction toward the opponent (sign of `opp_center_x − self_center_x`); stand-up exchanges are dominantly horizontal.
- **Lead foot** = the ankle (COCO 15 left / 16 right) projecting further along the forward axis. Left foot leads → orthodox; right foot leads → southpaw.
- **Visibility-gated:** ankles below `min_vis` are dropped; fall back to knees (13/14) → hips (11/12) if an ankle is occluded; abstain if too few visible.
- **Abstain (`ambiguous`)** when the lead/rear ankle separation along the forward axis is below threshold (square stance) — real switches pass through this square moment, so abstaining there is correct.
- **Tunable params (named variables):** `min_vis=0.5`, separation threshold.

**DO NOT re-add multi-joint orientation voting (shoulders/hips). It was tried and it FAILED — see below and DECISIONS.**

**Rejected — margin-driven multi-joint orientation rule (shoulders + hips + ankles):**
Built to fix camera-angle errors by voting on torso blading, weighted by geometric separation. **Overfit to the validation clip and did not work on real footage.** Tuning it to pass the two sample frames did not generalize: on broadcast footage the shoulder/hip signals collapse under side-on angles into pseudo-label noise, and margin-normalizing that noise produced confident-wrong calls and excessive `ambiguous` output. The added joints hurt more than they helped. Reverted to the ankle-only lead-foot rule, which is the validated, working method.

**Smoothing & aggregation:**
- `smooth_win` (sliding majority vote) removes single-frame flicker. Small + odd (current: 9 frames ≈ 0.3 s @30fps).
- `min_switch` = frames a new stance must *hold* to count as an official switch (current: 30 frames ≈ 1 s @30fps). Separate knob from smoothing — controls the switch *event* count, not per-frame labels.
- Output per fighter: `% orthodox / southpaw / ambiguous`, confirmed switch count, and a switch timeline.

## 6. Tech stack
- Python 3.10+ (local dev env runs 3.13 — watch for occasional lib lag, esp. Phase 3 torch/ST-GCN deps)
- `ultralytics` ≥ 8.3 (YOLO detection + pose, + built-in tracking)
- `opencv-python`, `numpy`
- **Tracker:** BoT-SORT via Ultralytics `model.track()` (built-in). See DECISIONS.
- Action-recognition + skill-tree visualization libs: TBD (Phase 3/4).
- **Training compute:** Google Colab free tier, Tesla T4. Weights stored in Google Drive.
- **Inference compute:** local M2 Mac (CPU or MPS). No GPU needed for short clips. Fall back to `device="cpu"` if MPS errors.

## 7. Success criteria
- **Phase 1 (MVP):** detector + pose model trained on V2, running on a sample clip with overlaid skeletons and stable per-frame output. ✅ **DONE.**
- **Phase 2:** rule-based stance working end-to-end on video. ✅ **DONE.**
- **Phase 3:** strike recognition on a small self-labeled set, MVP classes only.
- **Phase 4:** aggregation + a basic skill-tree visualization for one full fight.
- Validate strike counts against published UFC significant-strike stats as a free sanity check.

## 8. Coding conventions
- Concise, minimal comments, no filler.
- Ask before scaffolding large multi-file code.
- Prefer small testable scripts per stage over a monolith early on.
- Expose tunable parameters as explicit named variables, never hardcoded.
- Validate any new rule on the two sample label frames (`0000`, `0053`) before running on video.
